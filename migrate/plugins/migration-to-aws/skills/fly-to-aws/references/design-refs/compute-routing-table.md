# Compute Routing Table

Routing keys are **fly.toml/code signals × Clarify-confirmed intent**. Signals detect; intent is confirmed for every routing-relevant semantic. Layers evaluate in order per process group; **first match wins**. Order rule: specialized/exclusionary workload classes route before platform preference and scaling shape.

## Layer-5 Fine Print

min_machines_running applies to the primary region only, and fly-proxy never creates machines (no scale-out) — the Clarify wording must not imply the Fly config expressed multi-region floors or autoscaling intent that AWS target sizing then has to honor.

## Layer 0 Flow (Detection → Confirmation → Routing)

1. Discover greps the repo per process group and writes `agent_candidate: true` with the evidence list into the inventory. Detection is conservative — when unsure, omit (agent-advisor's rule: a wrong signal silently biases downstream).
2. Clarify presents each flagged group with its evidence and asks for confirmation (AskUserQuestion). The user can also volunteer an un-flagged group as an agent.
3. Only confirmed groups route to agent-advisor. The handoff is an offer, not automatic: "Group `X` is an AI agent — its runtime deserves the full scoring pass (AgentCore vs MicroVMs vs ECS/EKS/Lambda). Run it now? The rest of your resources continue in this migration flow either way."
4. On accept: the agent-advisor scoring runs for that group (see §6 for the mechanics); its verdict is recorded in `aws-design.json` as that group's compute target with `decided_by: "agent-advisor"`. On decline: the group falls through to layers 1–5 like any generic service, with a note that scoring was offered and declined.

## Forbidden Targets

Never route to: AWS App Runner (closed to new customers 2026-04-30), Copilot CLI (EOL), Elastic Beanstalk.
